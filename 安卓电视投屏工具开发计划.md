# 安卓电视投屏工具开发计划文档

## 1. 项目概述

### 1.1 项目目标

开发一款运行在 Android TV / 安卓电视盒子上的投屏接收端应用，使安卓手机、苹果手机、Windows 电脑、Mac 电脑可以将屏幕画面、音频或应用内容投屏到安装该应用的电视设备上。

### 1.2 产品定位

该应用定位为一个多协议投屏接收工具，核心目标是让电视具备类似以下能力：

- Android 手机投屏到电视
- iPhone / iPad 投屏到电视
- Windows 电脑投屏到电视
- Mac 电脑投屏到电视
- 支持屏幕镜像、视频播放、音频同步、设备发现和连接管理

### 1.3 目标运行平台

电视端应用：

- Android TV
- Google TV
- 安卓电视盒子
- 普通安卓系统电视设备

建议最低系统版本：

- Android 7.0 及以上
- 推荐 Android 9.0 及以上

## 2. 需求分析

### 2.1 核心功能需求

| 功能模块 | 描述 | 优先级 |
|---|---|---|
| 设备发现 | 手机或电脑能够在局域网中发现电视端应用 | 高 |
| 屏幕镜像 | 将发送端屏幕实时显示到电视 | 高 |
| 音频播放 | 同步播放发送端音频 | 高 |
| 视频流接收 | 接收 H.264 / H.265 视频流并解码播放 | 高 |
| 多协议支持 | 支持 Android、iOS、Windows、macOS 常见投屏协议 | 高 |
| 连接管理 | 显示连接状态、断开连接、重连 | 高 |
| 横竖屏适配 | 自动适配发送端屏幕比例 | 中 |
| 画质调节 | 支持分辨率、码率、帧率配置 | 中 |
| 投屏鉴权 | 支持连接确认、PIN 码或密码 | 中 |
| 后台常驻服务 | 电视端应用可持续等待连接 | 中 |
| 日志与诊断 | 便于调试网络、协议和播放问题 | 中 |

### 2.2 支持设备需求

#### Android 手机

目标能力：

- 支持安卓手机屏幕镜像
- 支持安卓应用内容投屏
- 支持音视频同步

可选方案：

1. Miracast / Wi-Fi Display
2. Google Cast 协议兼容方案
3. 自定义发送端 App + MediaProjection
4. WebRTC / RTSP 自定义投屏协议

说明：

- 原生 Miracast 接收端在 Android 应用层实现难度较高，部分能力受系统权限限制。
- 如果希望稳定可控，建议先实现“自定义发送端 App + 电视接收端”的方案。

#### 苹果手机 / iPad

目标能力：

- 支持 iOS 屏幕镜像
- 支持从系统控制中心选择电视设备进行投屏

可选方案：

1. AirPlay Receiver 协议兼容实现
2. 自定义 iOS App + WebRTC / RTSP
3. DLNA，仅适合媒体文件播放，不适合完整屏幕镜像

说明：

- AirPlay 是苹果私有协议，实现完整兼容存在技术和合规风险。
- 第一阶段可以先支持 DLNA / 自定义 App，后续再评估 AirPlay 兼容性。

#### Windows 电脑

目标能力：

- 支持桌面投屏
- 支持应用窗口投屏
- 支持声音同步

可选方案：

1. Miracast 接收兼容
2. Windows 自定义发送端程序
3. 浏览器 WebRTC 投屏
4. RTSP / RTP / WebSocket 视频流

建议方案：

- 优先开发 Windows 自定义发送端程序。
- 使用桌面采集 + H.264 编码 + WebRTC / RTSP 推流到电视。

#### Mac 电脑

目标能力：

- 支持 macOS 桌面投屏
- 支持应用窗口投屏
- 支持声音同步

可选方案：

1. AirPlay Receiver 协议兼容
2. macOS 自定义发送端程序
3. 浏览器 WebRTC 投屏

建议方案：

- 短期使用浏览器 WebRTC 或 macOS 自定义发送端。
- 长期再研究 AirPlay 兼容。

## 3. 技术路线选择

### 3.1 总体架构建议

建议采用“电视接收端 + 多平台发送端 + 自定义协议为主 + 原生协议逐步兼容”的架构。

原因：

- Miracast 和 AirPlay 都存在协议复杂、系统限制和兼容性问题。
- 直接做完整原生协议接收端，开发成本和风险较高。
- 自定义协议可以优先保证核心功能可用，再逐步兼容系统原生投屏入口。

### 3.2 推荐技术架构

```text
Android 手机发送端        \
                          \
Windows 发送端  --------->  Android TV 接收端应用
                          /
Mac 发送端      ---------/

iPhone 发送端  ----------/
```

电视端主要负责：

- 局域网设备广播
- 接收连接请求
- 接收视频流 / 音频流
- 解码播放
- 显示连接状态
- 控制连接生命周期

发送端主要负责：

- 屏幕采集
- 音频采集
- 编码压缩
- 网络推流
- 连接电视端

## 4. 协议方案设计

### 4.1 第一阶段推荐协议：WebRTC

#### 优点

- 低延迟
- 支持音视频同步
- 跨平台能力强
- Android、iOS、Windows、macOS、浏览器均可支持
- 有成熟开源库
- 支持 NAT 穿透，后续可扩展远程投屏

#### 缺点

- 集成复杂度中等
- 需要信令服务或局域网信令机制
- 对编解码和设备性能有要求

#### 适用场景

- 局域网投屏
- 实时屏幕镜像
- 游戏、会议、演示等低延迟场景

### 4.2 第二阶段协议：RTSP / RTP

#### 优点

- 实现相对直接
- 适合局域网音视频流传输
- Android 端可以使用 MediaCodec 解码

#### 缺点

- 延迟通常高于 WebRTC
- 跨平台客户端需要单独开发
- 浏览器支持较弱

### 4.3 第三阶段协议：DLNA

#### 优点

- 适合视频、音乐、图片投放
- 很多手机和电脑软件支持
- 实现成本低于 AirPlay / Miracast

#### 缺点

- 不适合实时屏幕镜像
- 更偏媒体文件投放

### 4.4 第四阶段协议：AirPlay / Miracast 兼容

#### AirPlay

- 目标：让 iPhone / Mac 可以直接在系统投屏列表中发现电视。
- 难点：协议私有、兼容性复杂、可能涉及授权问题。

#### Miracast

- 目标：让 Android / Windows 可以通过系统无线显示投屏到电视。
- 难点：Android 应用层不容易完整实现 Miracast Sink，可能需要系统级权限或底层支持。

## 7. 关键技术难点

### 7.1 原生协议兼容难度

#### Miracast

Miracast 接收端通常需要系统级 Wi-Fi Display Sink 支持，普通 Android 应用很难完整实现系统级 Miracast 接收能力。

应对策略：

- 第一阶段不强依赖系统级 Miracast。
- 先使用自定义 Android 发送端完成稳定投屏。
- 后续评估特定电视设备是否开放 Wi-Fi Display Sink 能力。

#### AirPlay

AirPlay 是苹果私有协议，完整实现存在兼容和法律风险。

应对策略：

- 第一阶段优先支持自定义 iOS 发送端。
- 后续调研开源 AirPlay Receiver 实现，如 RPiPlay、UxPlay 等。
- 明确商用合规边界。

### 7.2 音视频低延迟

目标指标：

- 普通演示：500ms 以下
- 视频播放：300ms - 800ms 可接受
- 游戏投屏：100ms - 200ms 以内较理想

优化方向：

- 使用硬件编码 / 硬件解码
- 控制 GOP 间隔
- 减少缓冲队列
- 使用 UDP / WebRTC
- 根据网络质量动态调整码率

### 7.3 音视频同步

需要处理：

- 视频帧时间戳
- 音频采样时间戳
- 网络抖动
- 播放缓冲
- 丢帧策略

### 7.4 电视设备兼容性

安卓电视设备差异较大，需要兼容：

- 不同 CPU 架构：ARMv7、ARM64、x86
- 不同 Android 版本
- 不同厂商系统权限限制
- 不同硬解码能力

| AirPlay 协议私有且兼容复杂 | 高 | 初期使用自定义 iOS App，后续谨慎兼容 |
| 低延迟投屏性能不足 | 中 | 使用 WebRTC、硬编硬解、动态码率 |
| 电视设备兼容性差 | 中 | 建立测试设备矩阵 |
| 音视频同步困难 | 中 | 基于时间戳同步和缓冲控制 |
| iOS 后台屏幕采集限制 | 中 | 使用 ReplayKit Broadcast Extension |

### 11.2 合规风险

| 风险 | 说明 | 应对策略 |
|---|---|---|
| AirPlay 商标和协议合规 | AirPlay 是苹果生态协议 | 避免使用相关商标，评估授权问题 |
| Miracast 商标合规 | Miracast 是 Wi-Fi Alliance 认证名称 | 商用前评估认证要求 |
| 第三方开源协议 | 部分库可能 GPL 限制 | 优先选择 Apache / MIT / BSD 协议库 |

- 请求屏幕录制权限

## 15. 总结

该项目的核心难点不在于单纯播放视频，而在于跨平台屏幕采集、编码传输、协议兼容、低延迟播放和设备兼容性。

推荐开发策略：

1. 先做自定义协议，保证 Android 手机、Windows、Mac 可以稳定投屏。
2. 再支持 iOS 自定义投屏。
3. 然后补充 DLNA 媒体投放。
4. 最后再评估 AirPlay 和 Miracast 原生兼容。

这样可以在较短时间内做出可用产品，同时避免一开始就陷入 AirPlay / Miracast 协议兼容的高风险工作。
